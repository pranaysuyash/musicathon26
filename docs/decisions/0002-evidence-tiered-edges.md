# Decision 0002 — Evidence-tiered graph edges

**Date:** 2026-06-16
**Status:** Active
**Owner:** VerseSignal agent

## Decision

Every `graph_edges` row carries **four pieces of metadata** in
addition to the `src_id` / `dst_id` / `edge_type` triple:

1. `weight` (0..1+ — strength of the connection)
2. `confidence` (0..1 — certainty of the connection)
3. `source_api` (musixmatch | songstats | gliner | spacy | embedding | llm | lexicon | cyanite | musicbrainz | wikidata | human | hybrid)
4. `model_version` (e.g., `sentence-transformers/all-MiniLM-L6-v2`, `urchade/gliner_medium-v2.1`, `lexicon_v1`)
5. `explanation` (human-readable "why")

And every edge references ≥1 row in the `evidence` table, each
carrying the **type** of evidence (lyric_line | lyric_term |
chart_entry | metadata_credit | mood_score | event_date_overlap |
entity_match | embedding_similarity | collaboration_credit), the
**literal value**, the **source API**, and a per-row **confidence**.

## Context

The naive graph design is `node -[label]-> node`. That's not a
knowledge graph; it's a decoration. The first version of
VerseSignal's UI (sketched before this decision) had edges with
only a `type` column. The demo would have been an attractive
visualisation with no defensible answer to "why is this connected?"

Per motto_v3 §0.11 (customer-facing claims) and §0.14 (operator
workflow), the system must be able to explain its own state.

## Options considered

| Option | Pros | Cons | Verdict |
|---|---|---|---|
| Edge with only `type` | Simplest | No provenance; no way to defend or debug | Rejected |
| Edge with `weight` only | Common in academic graphs | Doesn't separate strength from certainty (a strong guess ≠ a weak fact) | Rejected |
| Edge with weight + confidence + source + version + explanation | Full provenance; explainable; auditable | Schema heavier; one extra join for every click | **Chosen** |
| Edge with full provenance + per-evidence rows in separate table | Strongest; supports partial evidence (e.g., "3 lyric lines + 1 chart entry + 1 mood match" all contributing) | Heaviest schema | **Chosen** for product-critical edges (`associated_with_event`); lighter for stub edges (`performed_by`, `charted_in`) |

## Chosen path

Schema (in `scripts/schema.sql`):

```sql
CREATE TABLE graph_edges (
  id            TEXT PRIMARY KEY,
  src_id        TEXT NOT NULL REFERENCES graph_nodes(id),
  dst_id        TEXT NOT NULL REFERENCES graph_nodes(id),
  edge_type     TEXT NOT NULL,
  weight        REAL NOT NULL,        -- 0..1+; 1.0 = strong signal
  confidence    REAL NOT NULL,        -- 0..1; 1.0 = certain
  evidence_ids_json TEXT,             -- JSON array of evidence.id
  source_api    TEXT NOT NULL,        -- which API/model produced this
  model_version TEXT,
  explanation   TEXT,
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE evidence (
  id            TEXT PRIMARY KEY,
  edge_id       TEXT NOT NULL REFERENCES graph_edges(id),
  evidence_type TEXT NOT NULL,
  value         TEXT NOT NULL,
  source        TEXT NOT NULL,
  confidence    REAL NOT NULL,
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);
```

The UI (`components/evidence/evidence-drawer.tsx`) renders every
piece when an edge is clicked.

## Why this path

- A user clicking an edge sees: *weight, confidence, source API,
  model version, explanation, and a list of evidence rows with
  their own types and confidence*. They can challenge any link.
- A future agent can audit the graph: which edges were produced
  by which model, with what confidence, on which dataset.
- A/B testing models becomes a SQL diff: `SELECT edge_type,
  source_api, AVG(weight), AVG(confidence) FROM graph_edges
  GROUP BY edge_type, source_api`.
- The 11-dim audit (per 0.6) can check "every edge has evidence"
  as a single SQL query.

## Tradeoffs

- Schema weight: ~30% more storage than a flat-edge design.
  Acceptable for a 5,000-edge graph; would need denormalization at
  5M+ edges.
- Click latency: the evidence drawer triggers a second
  `getEvidenceForEdge` query. Currently synchronous, fast
  (single SQL lookup on indexed `edge_id`).
- Stale model versions: if a model is re-trained, old edges still
  reference the old version. **This is the correct behavior** —
  reproducibility requires we know which model produced each edge.

## Risks

- **Explanation quality:** LLM-style "explanation" rows are
  generated by the enrich pipeline and may hallucinate. We
  mitigate by always keeping raw evidence rows; explanations
  must reference real evidence IDs, not invent them.
- **Confidence inflation:** if we always set confidence=0.9 for
  the first version of every edge, the graph becomes
  indistinguishable from a fully-certain graph. We use
  `min(1.0, 0.5 + 0.05 * len(matched_terms))` for event links;
  similar formulas for other edge types. Documented in the
  enrich script.

## Validation plan

- [x] 2,231 edges × ≥1 evidence row = ≥2,231 evidence rows
      (actual: 5,273, because event-linked edges carry
      multiple evidence rows: lyric lines + matched terms +
      date overlap)
- [x] Every edge has `source_api` set (SQL: `SELECT COUNT(*)
      FROM graph_edges WHERE source_api IS NULL OR
      source_api = ''` → 0)
- [x] Every `associated_with_event` edge has ≥2 evidence rows
      (the strictest class)

## What would cause this decision to be revisited

- Graph exceeds ~50k edges and the JOIN becomes slow
- Users complain that the evidence drawer is "too much detail"
  (in which case we add a "simple" view alongside the evidence
  drawer, not replace it)
- A new edge type has zero evidence possible (e.g., a derived
  "transitive" edge); then we drop the requirement for that
  type and document the exception
